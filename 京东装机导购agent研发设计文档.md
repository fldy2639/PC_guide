# 京东装机导购 Agent 研发设计文档

## 0. 文档信息

| 项目 | 内容 |
|---|---|
| 项目名称 | 京东装机导购 Agent |
| 文档类型 | 研发可执行详细设计文档 |
| 当前版本 | V1.0 草案 |
| 目标平台 | 京东网页端导购场景 |
| 目标用户 | 装机入门用户、具备一定 DIY 知识的用户 |
| 核心目标 | 提高装机产品转化 |
| 首版数据策略 | 模拟数据 + 预留京东商品数据爬取链路 |
| 首版输出策略 | 输出 1 套最终装机方案，内部每个品类召回 Top3 候选 |

## 1. 项目背景与目标

### 1.1 项目背景

装机类商品具有高客单价、高决策成本、高兼容性要求的特点。用户在京东购买装机配件时，通常会遇到以下问题：

1. 不清楚不同配件之间是否兼容。
2. 不知道预算应该如何分配到 CPU、显卡、主板、内存、显示器等品类。
3. 面对大量 SKU 时，难以判断哪款更适合自己的使用场景。
4. 对性能、外观、价格之间的取舍缺乏明确判断。
5. 即使有明确偏好，也需要人工反复搜索、比价和确认。

因此，本项目希望在京东网页端构建一个装机导购 Agent，基于用户自然语言输入，自动理解需求、召回候选配件、完成兼容性和风险校验，并输出一套可购买的装机方案。

### 1.2 项目目标

首版项目目标如下：

1. 支持用户通过自然语言描述装机需求。
2. 自动抽取预算、用途、性能、外观、价格、品牌、指定品类、指定型号等约束。
3. 根据需求约束生成性能、价格、外观及其他因素的权重。
4. 在每个品类内部召回 Top3 候选配件。
5. 从候选配件中选择一套最终装机方案。
6. 对方案进行兼容性、预算、性能、风险校验。
7. 输出一套包含价格、推荐理由、风险提示和京东购买入口示意的装机方案。
8. 为后续接入真实京东商品数据、购物车、一键购买、追问模块和效果图渲染模块预留接口。

### 1.3 非目标范围

V1 阶段暂不实现以下能力：

1. 不做实时京东全站搜索与实时价格抓取。
2. 不做复杂库存、毛利、活动、商家策略排序。
3. 不做多轮复杂问答推理，只支持最多一次关键缺失信息追问。
4. 不直接生成真实购物车，仅输出京东购买链接占位或示意入口。
5. 不强依赖完整结构化商品参数，首版允许通过名称解析和规则库进行粗粒度兼容性校验。
6. 不强制输出显示器；当用户明确只要主机时，不推荐显示器。

## 2. 用户与使用场景

### 2.1 目标用户

| 用户类型 | 特征 | 典型诉求 |
|---|---|---|
| 装机入门用户 | 对硬件参数不熟悉，知道预算和用途 | 希望系统直接给出靠谱方案 |
| 有一定知识的用户 | 能指定 CPU、显卡、品牌、机箱风格等 | 希望系统在尊重偏好的基础上补齐整套配置 |
| 京东网页访客 | 正在浏览装机、显卡、CPU、整机或 DIY 配件页面 | 希望快速完成选购决策 |

### 2.2 典型输入示例

#### 示例 1：规范输入

```text
我想配一台 6000-8000 元的主机，用于畅玩 3A 大作，最好白色海景房，不需要显示器。
```

#### 示例 2：指定某一品类

```text
我想配一台 9000 元左右的电脑，显卡想用 RTX 4070 SUPER，主要玩 2K 游戏，外观尽量简洁。
```

处理原则：

- 用户指定某一品类想要某一款时，该配件作为硬约束优先填入。
- 后续 Agent 需要围绕该指定配件补齐其他品类。
- 如果指定配件导致预算、兼容性或风险问题，需要在检查阶段明确提示，并给出替代方案。

#### 示例 3：信息缺失

```text
我想配一台打游戏的电脑。
```

处理原则：

- V1 支持最多一次追问。
- 优先追问预算和是否需要显示器。
- 后续预留可扩展追问模块接口。

## 3. V1 装机品类范围

V1 支持最多十个品类：

| 序号 | 品类 | 是否必选 | 说明 |
|---:|---|---|---|
| 1 | CPU / 处理器 | 是 | 核心计算性能来源 |
| 2 | 散热 | 是 | 风冷或水冷 |
| 3 | 主板 | 是 | 需与 CPU、内存、机箱兼容 |
| 4 | 显卡 | 视需求 | 游戏/AI/渲染场景优先；办公场景可选核显方案 |
| 5 | 内存 | 是 | 需匹配主板内存类型 |
| 6 | 硬盘 | 是 | 默认 SSD |
| 7 | 机箱 | 是 | 需匹配主板版型、显卡长度、散热规格 |
| 8 | 电源 | 是 | 需满足整机功耗和安全余量 |
| 9 | 风扇 | 可选 | 颜值、散热、风道优化 |
| 10 | 显示器 | 可选 | 用户需要整套桌面方案时推荐 |

### 3.1 显示器推荐规则

1. 用户明确说“只要主机”“不含显示器”时，不推荐显示器。
2. 用户明确说“含显示器”“整套”“显示器也要”时，推荐显示器。
3. 用户未说明时：
   - 如果预算明显包含显示器，例如“整套 8000 元”，可以推荐显示器。
   - 如果表达为“主机 8000 元”，不推荐显示器。
   - 如果不明确，V1 可在追问中确认。

## 4. 总体 Agent 架构

### 4.1 总体流程

```text
用户自然语言输入
        ↓
需求理解 Agent
        ↓
需求约束列表 + 权重 + 是否需要追问
        ↓
配件选择 Agent
        ↓
每个品类 Top3 候选配件
        ↓
检查与补充 Agent
        ↓
兼容性校验 / 预算校验 / 风险校验 / 替代方案
        ↓
方案输出 Agent
        ↓
最终装机方案表格 + 推荐理由 + 风险提示 + 京东购买入口示意
        ↓
效果图渲染接口，V1 预留
```

### 4.2 Agent 模块划分

| Agent | 输入 | 输出 | 主要职责 |
|---|---|---|---|
| 需求理解 Agent | 用户自然语言 | 结构化需求、约束、权重、追问判断 | 理解预算、用途、偏好、指定配件 |
| 追问 Agent，V1 简化 | 缺失需求项 | 一次追问问题 | 当关键信息缺失时询问用户 |
| 配件选择 Agent | 需求约束、权重、商品库 | 每品类 Top3 候选 | 从商品池召回候选配件 |
| 检查与补充 Agent | 候选配件组合 | 最终方案或失败原因 | 做兼容性、预算、风险和需求满足校验 |
| 方案输出 Agent | 最终方案 | 用户可读方案 | 输出表格、理由、风险、购买入口 |
| 效果图渲染 Agent，预留 | 配件名称、机箱风格 | 渲染图 | 后续用于生成装机效果图 |

## 5. 需求理解 Agent 设计

### 5.1 输入

```json
{
  "user_query": "我想配一台 6000-8000 元的主机，用于畅玩 3A 大作，最好白色海景房，不需要显示器。",
  "conversation_context": [],
  "version": "v1"
}
```

### 5.2 输出

```json
{
  "need_clarification": false,
  "clarification_question": null,
  "requirements": {
    "budget": {
      "min": 6000,
      "max": 8000,
      "currency": "CNY",
      "strictness": "medium"
    },
    "usage": ["3A游戏"],
    "performance": {
      "target_games": ["3A大作"],
      "resolution": null,
      "fps_target": null,
      "priority": "high"
    },
    "appearance": {
      "color": "white",
      "style": "海景房",
      "rgb_preference": null,
      "priority": "medium"
    },
    "price": {
      "priority": "medium",
      "allow_over_budget": true,
      "over_budget_policy": "within_5_percent_ask_user"
    },
    "display": {
      "need_monitor": false
    },
    "specified_parts": [],
    "brand_preferences": [],
    "avoid_preferences": [],
    "other_constraints": []
  },
  "weights": {
    "performance": 0.5,
    "price": 0.3,
    "appearance": 0.2,
    "other": 0.0
  },
  "explanation": "用户明确要求 6000-8000 元预算、3A 游戏和白色海景房外观，因此优先保证游戏性能，其次控制预算，再兼顾白色外观。"
}
```

### 5.3 需求约束字段

| 字段 | 类型 | 说明 |
|---|---|---|
| budget | object | 预算范围 |
| usage | array | 用途，如 3A 游戏、办公、剪辑、AI、本地模型等 |
| performance | object | 性能目标，如分辨率、帧率、游戏类型 |
| appearance | object | 外观偏好，如白色、黑色、海景房、RGB、静音、小机箱 |
| price | object | 价格敏感度、是否允许小幅超预算 |
| display | object | 是否需要显示器 |
| specified_parts | array | 用户指定品类或型号 |
| brand_preferences | array | 品牌偏好 |
| avoid_preferences | array | 避免品牌、避免二手、避免水冷等 |
| other_constraints | array | 其他约束 |

### 5.4 用户指定配件处理规则

当用户明确指定某一品类或型号时，将其写入 `specified_parts`：

```json
{
  "category": "GPU",
  "user_text": "RTX 4070 SUPER",
  "match_mode": "fuzzy",
  "constraint_level": "hard"
}
```

处理原则：

1. 指定配件优先进入候选池。
2. 如商品库中存在多个近似 SKU，召回最匹配的 Top3。
3. 如果指定配件与预算冲突，检查阶段提示冲突并给出替代建议。
4. 如果指定配件与其他硬件不兼容，检查阶段不得直接输出为最终方案，必须提示原因。

### 5.5 权重生成规则

默认权重根据用户输入中的约束强度动态生成。

#### 5.5.1 权重维度

| 权重项 | 说明 |
|---|---|
| performance | 性能优先级 |
| price | 价格/预算优先级 |
| appearance | 外观优先级 |
| other | 其他因素，如品牌、静音、小机箱、生产力等 |

#### 5.5.2 示例权重

| 用户表达 | 性能 | 价格 | 外观 | 其他 |
|---|---:|---:|---:|---:|
| “畅玩 3A，大概 8000” | 0.55 | 0.35 | 0.10 | 0.00 |
| “预算不能超 6000，能玩网游” | 0.35 | 0.55 | 0.10 | 0.00 |
| “白色海景房，颜值高，能玩 2K” | 0.40 | 0.25 | 0.35 | 0.00 |
| “我要安静、小机箱、办公剪辑” | 0.35 | 0.25 | 0.15 | 0.25 |

#### 5.5.3 用户显式指定权重

用户表达如：

```text
预算可以超一点，但显卡一定要好。
```

处理方式：

- 提升性能权重。
- 降低价格严格程度。
- 将显卡作为关键品类提升召回和排序优先级。

### 5.6 追问机制

V1 支持最多一次追问。

#### 5.6.1 需要追问的情况

| 情况 | 示例 | 追问内容 |
|---|---|---|
| 缺预算 | “我想配一台打游戏的电脑” | “预算大概是多少？是否需要显示器？” |
| 缺用途 | “我想配一台 8000 的电脑” | “主要用于游戏、办公、剪辑还是其他？” |
| 显示器范围不明确 | “整套 8000” | “这个预算是否需要包含显示器？” |
| 关键约束冲突 | “4000 元 4K 高画质 3A” | “该目标预算偏紧，是否接受降低到 1080P/2K 或提高预算？” |

#### 5.6.2 追问接口预留

```json
{
  "need_clarification": true,
  "missing_fields": ["budget", "need_monitor"],
  "clarification_question": "请问你的预算大概是多少？这个预算是否需要包含显示器？",
  "next_action": "wait_user_reply"
}
```

后续可扩展为多轮追问模块：

```text
用户输入 → 需求完整度评分 → 缺失字段识别 → 追问策略选择 → 用户回答融合 → 重新生成需求约束
```

## 6. 商品数据设计

### 6.1 V1 模拟数据字段

V1 只要求包含配件名称、价格、类别。

```json
{
  "sku_id": "mock_gpu_001",
  "category": "GPU",
  "name": "华硕 ATS RTX 4070 SUPER O12G 显卡",
  "price": 5050,
  "jd_url": "https://item.jd.com/mock_gpu_001.html"
}
```

### 6.2 V1 商品表结构

| 字段 | 类型 | 是否必需 | 说明 |
|---|---|---|---|
| sku_id | string | 是 | 商品唯一 ID |
| category | string | 是 | 商品品类 |
| name | string | 是 | 商品名称 |
| price | number | 是 | 当前参考价 |
| jd_url | string | 否 | 京东购买入口示意 |

### 6.3 V2 扩展字段

为支持更准确的兼容性校验和排序，V2 建议补充以下字段：

```json
{
  "sku_id": "gpu_4070s_001",
  "category": "GPU",
  "name": "华硕 ATS RTX 4070 SUPER O12G 显卡",
  "price": 5050,
  "brand": "华硕",
  "series": "RTX 4070 SUPER",
  "performance_score": 88,
  "appearance_tags": ["黑色", "三风扇"],
  "power_watt": 220,
  "length_mm": 305,
  "stock_status": "in_stock",
  "is_jd_self_operated": true,
  "gross_margin_level": "medium",
  "jd_url": "https://item.jd.com/xxx.html"
}
```

### 6.4 京东数据链路设计

本项目采用“模拟数据 + 真实数据链路预留”的设计。

#### V1：模拟数据

- 人工构建 JSON/CSV 商品库。
- 字段只包含：类别、名称、价格、购买链接占位。
- 兼容性校验依赖名称解析和基础规则。

#### V2：离线爬取数据

- 爬取京东搜索页或商品页。
- 定期生成离线商品库。
- 增加品牌、价格、评价、销量、自营标记、库存、规格等字段。
- 排序时引入京东自营优先、转化率、价格竞争力等因子。

#### V3：实时商品能力

- 接入实时价格、库存、活动、优惠券、商品链接。
- 支持一键加入购物车或生成购买清单。
- 支持根据库存和价格波动动态更新方案。

## 7. 配件选择 Agent 设计

### 7.1 输入

```json
{
  "requirements": {},
  "weights": {
    "performance": 0.5,
    "price": 0.3,
    "appearance": 0.2,
    "other": 0.0
  },
  "product_pool": []
}
```

### 7.2 输出

每个品类输出 Top3 候选。

```json
{
  "candidates": {
    "CPU": [
      {"sku_id": "cpu_001", "name": "Intel i5-13600KF", "price": 1531, "score": 0.88},
      {"sku_id": "cpu_002", "name": "Intel i5-14600KF", "price": 1780, "score": 0.84},
      {"sku_id": "cpu_003", "name": "AMD Ryzen 5 7600", "price": 1299, "score": 0.78}
    ],
    "GPU": [
      {"sku_id": "gpu_001", "name": "RTX 4070 SUPER", "price": 5050, "score": 0.91}
    ]
  }
}
```

### 7.3 召回策略

每个品类的 Top3 召回由以下因素综合决定：

```text
候选得分 = 性能匹配分 × 性能权重
        + 价格匹配分 × 价格权重
        + 外观匹配分 × 外观权重
        + 其他偏好分 × 其他权重
        + 指定配件加权
        + 京东自营/商业因子预留
        - 风险惩罚分
```

V1 暂不做复杂商业排序，但预留字段。

### 7.4 指定配件优先策略

如果用户指定某个品类：

1. 先在该品类商品池中做名称匹配。
2. 精确匹配优先，其次模糊匹配。
3. 指定配件进入该品类候选 Top3。
4. 如果该配件价格过高或存在兼容风险，仍进入候选池，但由检查 Agent 决定是否可进入最终方案。

### 7.5 品类预算分配建议

V1 可使用经验比例进行初始预算分配。

#### 游戏主机，不含显示器

| 品类 | 预算占比建议 |
|---|---:|
| 显卡 | 35%-50% |
| CPU | 15%-25% |
| 主板 | 8%-12% |
| 内存 | 6%-10% |
| 硬盘 | 6%-10% |
| 电源 | 5%-8% |
| 机箱 | 4%-8% |
| 散热 | 3%-6% |
| 风扇 | 0%-5% |

#### 含显示器方案

| 品类 | 预算占比建议 |
|---|---:|
| 显示器 | 15%-25% |
| 显卡 | 30%-40% |
| CPU | 12%-20% |
| 其他配件 | 25%-40% |

## 8. 检查与补充 Agent 设计

### 8.1 输入

```json
{
  "requirements": {},
  "weights": {},
  "candidates": {},
  "budget": {
    "min": 6000,
    "max": 8000
  }
}
```

### 8.2 输出

```json
{
  "status": "success",
  "final_build": [
    {"category": "CPU", "name": "Intel i5-13600KF", "price": 1531},
    {"category": "GPU", "name": "RTX 4070 SUPER", "price": 5050}
  ],
  "total_price": 7850,
  "budget_check": {
    "status": "within_budget",
    "over_budget_rate": 0
  },
  "compatibility_check": {
    "status": "pass",
    "warnings": []
  },
  "risk_check": {
    "status": "pass_with_notes",
    "warnings": ["建议选择 750W 以上金牌电源，保证后续升级余量。"]
  },
  "unmet_constraints": [],
  "alternative_suggestions": []
}
```

### 8.3 预算校验规则

| 情况 | 处理策略 |
|---|---|
| 总价 ≤ 预算上限 | 直接输出方案 |
| 超预算 ≤ 5% | 输出方案，并询问用户是否接受小幅超预算 |
| 超预算 5%-15% | 说明超预算原因，并给出降配建议 |
| 超预算 > 15% | 判定当前方案不满足预算，优先输出降配替代方案 |

### 8.4 兼容性校验规则

由于 V1 商品字段有限，采用“名称解析 + 规则库”的分阶段方案。

#### V1：名称解析 + 粗规则

可从商品名称中解析：

| 信息 | 示例 |
|---|---|
| CPU 型号 | i5-13600KF、R5 7600 |
| 主板芯片组 | B760、Z790、B650、A620 |
| 内存类型 | DDR4、DDR5 |
| 显卡型号 | RTX 4070 SUPER、RTX 4060 Ti |
| 电源功率 | 650W、750W、850W |
| 机箱规格 | ATX、MATX、ITX、支持 360 水冷 |
| 散热规格 | 风冷、240 水冷、360 水冷 |

#### V1 规则示例

| 校验项 | 规则 |
|---|---|
| CPU 与主板 | Intel 13/14 代优先匹配 B760/Z790；AMD AM5 优先匹配 A620/B650/X670 |
| 内存与主板 | DDR4 内存不得搭配 DDR5 主板，反之亦然 |
| 电源功率 | 根据 CPU + GPU 粗略功耗估算，建议保留 30% 左右余量 |
| 机箱与主板 | ATX 主板需 ATX 或兼容机箱；MATX 主板可装入多数 ATX/MATX 机箱 |
| 散热与机箱 | 360 水冷需机箱标注支持 360 冷排 |
| 显卡与机箱 | V1 可通过显卡长度字段缺失时给出风险提示；V2 增加尺寸校验 |

#### V2：结构化参数校验

后续数据应增加：

- CPU socket
- 主板 socket
- 主板芯片组
- 主板内存类型
- 主板版型
- 显卡长度
- 机箱显卡限长
- 散热器高度 / 冷排尺寸
- 电源额定功率
- 电源接口
- 商品库存与售后信息

### 8.5 风险搭配规避规则

以下情况不得直接作为“无风险方案”输出：

1. 高功耗 CPU + 高功耗显卡 + 低瓦数电源。
2. 高端 CPU 搭配明显供电较弱的低端主板。
3. K/KF 系列高功耗 CPU 搭配过弱散热。
4. 360 水冷搭配不支持 360 冷排的机箱。
5. DDR4 内存搭配 DDR5 主板。
6. 显卡可能过长但机箱参数缺失时，需要提示风险。
7. 预算明显不足但试图强行满足高分辨率高画质目标。
8. 低价未知品牌电源用于中高端显卡方案。

### 8.6 替代方案输出规则

当无法满足用户需求时，输出：

1. 无法满足的约束。
2. 原因说明。
3. 至少一个替代方向。

示例：

```json
{
  "status": "failed_with_alternative",
  "unmet_constraints": ["price", "performance"],
  "reason": "6000 元预算较难实现 4K 高画质 3A 游戏体验。",
  "alternative_suggestions": [
    "将目标调整为 2K 高画质游戏。",
    "将预算提升到 9000 元以上。",
```

## 14. 核心流程伪代码

```python
def recommend_pc_build(user_query: str):
    # 1. 理解需求
    parsed = requirement_agent.parse(user_query)

    # 2. 判断是否需要追问
    if parsed.need_clarification:
        return {
            "type": "clarification",
            "question": parsed.clarification_question
        }

    # 3. 加载商品池
    product_pool = product_service.load_products(version="v1_mock")

    # 4. 每个品类召回 Top3
    candidates = selection_agent.retrieve_top3(
        requirements=parsed.requirements,
        weights=parsed.weights,
        product_pool=product_pool
    )

    # 5. 校验并选择最终方案
    validation_result = validation_agent.validate_and_select(
        requirements=parsed.requirements,
        weights=parsed.weights,
        candidates=candidates
    )

    # 6. 如果无法满足，生成替代建议
    if validation_result.status == "failed_with_alternative":
        return output_agent.render_failed_with_alternative(
            parsed=parsed,
            validation_result=validation_result
        )

    # 7. 如果小幅超预算，提示用户确认
    if validation_result.status == "need_user_confirmation":
        return output_agent.render_need_confirmation(
            parsed=parsed,
            validation_result=validation_result
        )

    # 8. 输出最终方案
    return output_agent.render_success(
        parsed=parsed,
        validation_result=validation_result
    )
```

| 权重合理性评分 | 人工评估权重是否符合用户表达 |

### 16.2 方案质量指标

| 指标 | 说明 |
|---|---|
| 预算满足率 | 最终方案是否满足预算 |
| 兼容性通过率 | 是否无明显硬件兼容错误 |
| 风险搭配率 | 是否出现不合理电源、散热、主板搭配 |
| 用户偏好满足率 | 是否满足外观、品牌、指定配件等偏好 |
| 替代方案有效率 | 无法满足时，替代建议是否可执行 |

### 16.3 业务指标

| 指标 | 说明 |
|---|---|
| 点击购买入口率 | 用户点击方案中京东商品入口的比例 |
| 加购率 | 后续接入购物车后的加购比例 |
| 成交转化率 | 装机方案带来的下单转化 |
| 客单价 | 平均方案金额 |
| 方案接受率 | 用户是否认可推荐方案 |
| 二次追问率 | 用户对方案继续追问或修改的比例 |

## 19. 风险与边界

### 19.1 数据风险

| 风险 | 说明 | 应对 |
|---|---|---|
| 商品价格变化 | 京东价格可能动态变化 | 输出“参考价”，真实价格以下单页为准 |
| 商品库存变化 | 商品可能缺货 | V2 接入库存字段 |
| 爬取稳定性 | 页面结构变化导致爬取失败 | 优先设计离线更新和异常兜底 |

### 19.2 推荐风险

| 风险 | 说明 | 应对 |
|---|---|---|
| 兼容性误判 | V1 字段不足导致校验不完整 | 输出风险提示，V2 增加结构化参数 |
| 预算不准确 | 模拟价格与真实价格不一致 | 标注参考价 |
| 用户需求模糊 | 输入信息不足 | V1 最多追问一次，后续扩展追问模块 |
| 指定配件冲突 | 用户指定配件与预算或兼容性冲突 | 明确提示冲突并给替代方案 |

### 19.3 表达风险

| 风险 | 说明 | 应对 |
|---|---|---|
| 过度承诺性能 | 不同游戏、画质、驱动影响帧率 | 使用“适合”“预计”“建议”等表述 |
| 用户理解成本高 | 参数过多 | 默认用易懂语言，专业细节可放在说明中 |
| 商业排序影响信任 | 如果只按商业目标推荐，可能损害体验 | V1 以用户需求匹配为主，商业因子后置 |

## 24. 总结

V1 的核心不是一次性解决所有商品数据和兼容性问题，而是先建立稳定的 Agent 流程：

```text
理解需求 → 生成权重 → 召回候选 → 校验方案 → 输出推荐 → 预留购买与渲染接口
```

在该流程跑通后，再逐步接入真实京东商品数据、结构化硬件参数、商业排序、购物车转化和效果图渲染能力。这样可以先验证装机 Agent 的导购转化价值，再逐步提升推荐准确性和商业闭环能力。
